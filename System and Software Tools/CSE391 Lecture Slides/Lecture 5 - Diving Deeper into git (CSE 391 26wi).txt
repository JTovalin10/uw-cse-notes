pull & merge requests
In practice, we often use tools like GitHub or GitLab to help us merge:
1.
2.
3.
4.
5.

create a local branch & make some commits
push those commits + branch to remote (e.g. GitHub or GitLab repo)
open a pull request (GitHub) or merge request (GitLab)
collaborate with others (e.g. code review & comments), ﬁx conﬂicts
merge into main (or other branch)

While this means we don’t typically run git merge manually,
●
●

it’s important to know how it works – that’s what GitHub & GitLab do!
you may have to use it for tougher merge conﬂicts

more git: reverting
●

git revert: adds a new commit undoing the change
○
○

●

git reset & force-push: move HEAD to a previous commit
○
○

●

pro: does not alter the previous git history (easy to work with others)
con: does not alter the previous git history (does not hide information)
pro: alters the git history (removes sensitive information)
con: alters the git history (causes irreconcilable changes with collaborators)

our advice:
○
○

on main and other shared branches: do not alter the git history
on your own, private branch: do what you want, but ideally avoid force-pushing

more git: merge strategies
Many diﬀerent ways to merge – what we learned is called a “merge commit”.
Other options include:
●
●
●

git rebase: “reapply” branch commits to a diﬀerent base branch
“squash” merging: rewrite history to make all changes 1 commit
other: fast-forward, custom strategies (e.g. handling 3-way merges)

Mostly out of scope of this class, but very relevant in bigger projects!
