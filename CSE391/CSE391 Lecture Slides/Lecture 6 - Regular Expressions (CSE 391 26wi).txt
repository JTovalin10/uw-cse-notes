What are regular expressions?
Depends on who you ask, but broadly:
an approach to ﬁnd structured patterns within strings
●
●

in theoretical cs & linguistics, “regular expression” has a formal deﬁnition
in software engineering, used more loosely beyond this deﬁnition

Virtually all programming languages implement regular expressions:
●
●
●

e.g. Java’s String library methods, like split() (via the Pattern class)
e.g. Python’s re library, JavaScript’s RegExp object
e.g. grep’s “Extended Regular Expression” mode, which is what we’ll use!

Why regular expressions?
In short: many software engineering problems are pattern matching & string
search. Regexes let you avoid reinventing the wheel!

Regexes are everywhere. Some common applications:
●
●
●

search for text in a ﬁle/input stream (most of this lecture)
validate text to match a speciﬁc format (much of upcoming homework)
ﬁnd-and-replace text (next lecture + homework)

aside: standardizing “regular expression”
There are many diﬀerent “ﬂavours” of regular expressions!
●
●
●

we’ve just learned grep’s “basic/extended regexes” (~ GNU BRE & ERE)
another common (and more powerful) family is PCRE
(Perl-Compatible Regular Expressions)
some languages implement special metacharacters or features (e.g. Java)

So: be mindful with online resources (and use the right library)!
However, what you’ve learned in this class applies to all of the above :)

aside: regular expressions and theory
If you’ve taken CSE 311, you heard of regexes in the context of “regular
languages” and ﬁnite automata (DFAs & NFAs). These are the same idea!
In fact, many regex engines are implemented via DFAs (including grep and
Rust’s & Go’s regex libs). The author of the Rust library & ripgrep’s has a great
blog post diving much deeper into how regex libraries work.
(strictly speaking, most regex libraries support features that technically supersede regular languages.
This includes backreferences, the $ and ^ anchors, and a feature called “lookaheads” or “lookarounds”. These
features also make the worst-case performance of regexes worse!)

aside: limits of regular expressions
But, this also means that regular expressions suﬀer some weaknesses
(i.e., they cannot recognize irregular languages).
Non-exhaustive list of things (normal) regexes can’t do:
●
●
●

check if a string has “matching” or “balanced” parentheses (D1 Dyck lang)
look for a string of n a’s followed by n b’s (for any n >= 0)
checking for syntactically-valid (Java, Python, JavaScript) program

The proofs for these are fascinating (and a great application of theoretical CS)!
(note: regex extensions like recursive patterns solve these problems – but they’re not really regexes)
