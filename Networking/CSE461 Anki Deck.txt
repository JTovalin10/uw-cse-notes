What is the Nyquist Theorem and how do you apply it?	States the maximum data rate for a <b>noiseless</b> channel with bandwidth B transmitting signals with V discrete voltage levels.<br><b>Formula:</b> Max Rate = 2B · log₂(V) bps<br><b>Step-by-step application:</b><br>1. Identify bandwidth B in Hz<br>2. Identify number of signal levels V (e.g., V=2 for binary, V=4 for PAM-4)<br>3. Compute: Max Rate = 2 × B × log₂(V)<br><b>Example:</b> B=3 kHz, V=4 levels → 2 × 3000 × 2 = 12,000 bps<br><b>Key insight:</b> Nyquist tells you the limit imposed by the number of voltage levels — it assumes zero noise. Real channels are further limited by Shannon.
What is Shannon's Theorem and how does it set the absolute capacity ceiling?	Gives the maximum data rate for a channel <b>with additive noise</b>.<br><b>Formula:</b> C = B · log₂(1 + SNR) bps<br><b>Application:</b><br>1. Convert SNR_dB to linear: SNR = 10^(SNR_dB/10)<br>2. Compute: C = B × log₂(1 + SNR)<br><b>Example:</b> B=3 kHz, SNR=31 → C = 3000 × log₂(32) = 3000 × 5 = 15,000 bps<br><b>Why it's the absolute ceiling:</b> No encoding scheme can exceed Shannon capacity — it accounts for all noise regardless of signal levels.<br><b>Interaction with Nyquist:</b> Nyquist says V levels gives 2B·log₂V bps; Shannon says at a given SNR you can use at most log₂(1+SNR) bits/symbol. The tighter constraint applies.
What is the Signal-to-Noise Ratio (SNR) and how is it used?	The ratio of signal power to noise power on a channel, measured in decibels (dB).<br><b>Formula:</b> SNR_dB = 10 · log₁₀(Signal Power / Noise Power)<br>Linear SNR = 10^(SNR_dB / 10)<br>A higher SNR = cleaner signal = higher achievable data rate.<br>SNR is the key input to Shannon's formula: C = B · log₂(1 + SNR).
What is baud rate vs. bit rate?	<b>Baud rate:</b> Number of signal changes (symbols) per second.<br><b>Bit rate:</b> Number of bits transmitted per second.<br><b>Relationship:</b> Bit rate = Baud rate × log₂(V) where V = signal levels<br><b>Example:</b> Baud rate = 1200, V=4 levels (2 bits/symbol) → Bit rate = 2400 bps<br>Baud rate = bit rate only when each symbol encodes exactly 1 bit (V=2).
Walk through how NRZ, NRZI, Manchester, and 4B/5B encoding work bit by bit.	<b>NRZ:</b> 1=high, 0=low. Data 0110 → LOW HIGH HIGH LOW. Problem: long same-bit runs = no transitions = clock drift.<br><b>NRZI:</b> Transition on 1, no transition on 0. Data 0110 starting LOW: 0=LOW→LOW, 1=LOW→HIGH, 1=HIGH→LOW, 0=LOW→LOW. Fixes consecutive 1s; consecutive 0s still no transitions.<br><b>Manchester:</b> 0=high→low at midpoint; 1=low→high at midpoint. Always transitions → self-clocking. Data 01: [HIGH|LOW][LOW|HIGH]. Costs 2× bandwidth (50% efficiency).<br><b>4B/5B:</b> Maps each 4-bit nibble to a 5-bit code guaranteed no more than one leading 0 or two trailing 0s. E.g., 0000→11110, 0001→01001. 80% efficiency. Paired with NRZI to handle remaining 0-runs.
What are the four components of packet latency?	<b>Transmission delay:</b> Packet size / Link bandwidth — time to push all bits onto the wire<br><b>Propagation delay:</b> Distance / Speed of light in medium (~2×10⁸ m/s for fiber) — time for bit to travel from A to B<br><b>Processing delay:</b> Time for router to examine header (microseconds, usually negligible)<br><b>Queuing delay:</b> Time packet waits in router buffer (variable; dominated by congestion)<br><b>Total = Σ(all four) across all hops.</b><br><b>Typical dominance:</b> Propagation dominates on WAN; transmission dominates for large files on slow links; queuing dominates under congestion.
What is the bandwidth-delay product (BDP) and why does it matter?	BDP = Bandwidth × RTT (Round-Trip Time)<br>Represents the amount of data "in flight" in the pipe at any moment — the amount needed to keep the link fully utilized.<br><b>Example:</b> 100 Mbps link, 50 ms RTT → BDP = 100×10⁶ × 0.05 = 5,000,000 bits = 625 KB<br><b>Significance for TCP:</b> The congestion window (cwnd) must be ≥ BDP/2 to fully utilize the link. On satellite links (500ms RTT), cwnd must be huge — TCP's default 65 KB window is far too small.
Compare twisted pair, coaxial cable, fiber optic, and wireless.	<b>Twisted Pair:</b> Cheap, ubiquitous (Ethernet Cat5e/6, telephone). Limited by distance and EMI. Up to 10 Gbps short distances. Twisting reduces crosstalk.<br><b>Coaxial Cable:</b> Better shielding, higher bandwidth. Used in cable TV/broadband internet.<br><b>Fiber Optic:</b> Transmits light pulses. Terabits/sec, hundreds of km without repeaters, immune to EMI. Single-mode (narrow core, laser, long-haul) vs. multi-mode (wider core, LED, short-haul).<br><b>Wireless:</b> No physical medium; shared broadcast; subject to interference, fading, hidden terminal. Bandwidth shared across all users competing on the same frequency.
How does Byte Stuffing work step by step (sender and receiver)?	<b>Sentinel byte:</b> FLAG=0x7E (frame delimiter), ESC=0x7D.<br><b>Sender algorithm:</b><br>1. Begin frame with FLAG<br>2. For each data byte: if byte==FLAG → emit ESC, emit (FLAG XOR 0x20)=0x5E; if byte==ESC → emit ESC, emit (ESC XOR 0x20)=0x5D; else → emit byte unchanged<br>3. End frame with FLAG<br><b>Example:</b> Data=[0x7E, 0x41, 0x7D] → sent as: FLAG, 0x7D, 0x5E, 0x41, 0x7D, 0x5D, FLAG<br><b>Receiver algorithm:</b><br>1. Discard bytes until FLAG seen → start of frame<br>2. If next byte==ESC: discard ESC, XOR following byte with 0x20 to restore original<br>3. If next byte==FLAG: end of frame — pass accumulated bytes upward<br>4. Otherwise: accumulate byte<br><b>Used in:</b> PPP, HDLC
How does Bit Stuffing work step by step?	<b>Flag pattern:</b> 01111110 (0, six 1s, 0) marks frame boundaries.<br><b>Sender algorithm:</b><br>1. Transmit start flag: 01111110<br>2. Transmit data bits; after every five consecutive 1s in the data, automatically insert (stuff) a 0-bit<br><b>Example:</b> Data ...01111110... → transmitted as ...011111<b>0</b>10... (stuffed 0 inserted after five 1s)<br><b>Receiver algorithm:</b><br>1. On seeing five consecutive 1-bits: examine next bit<br>   • If 0 → discard it (stuffed); continue<br>   • If 1 → examine next: if 0 → frame end (valid flag 01111110); if 1 → ERROR (abort sequence)<br><b>Guarantee:</b> The flag pattern 01111110 can never appear in the data — every run of five 1s is broken by a stuffed 0.
How does CRC computation work? Give a complete example.	<b>Model:</b> Message M and generator G are polynomials. Append r zero bits (r = degree of G), divide by G, append remainder R.<br><b>Worked example:</b> M=1101, G=1011 (degree r=3)<br>1. Append 3 zeros: 1101000<br>2. XOR long division of 1101000 by 1011:<br>   1101000<br>  ÷1011: 1101 XOR 1011 = 0110; bring down 0 → 1100<br>   1100 XOR 1011 = 0111; bring down 0 → 1110<br>   1110 XOR 1011 = 0101; bring down 0 → remainder = 101<br>3. Transmitted frame = 1101 <b>101</b> (message + 3-bit remainder)<br><b>Receiver check:</b> Divide 1101101 by 1011 → remainder=000 → no error detected<br><b>Error detection capability of CRC:</b> All single-bit errors, all burst errors ≤ r bits, all odd-bit errors (if G has factor x+1).
Walk through Stop-and-Wait ARQ — sender and receiver state machines.	<b>Sender State A (Ready to Send):</b><br>• Get packet from layer above → transmit frame(seq=0) → start timer → go to State B<br><b>Sender State B (Waiting for ACK 0):</b><br>• Receive valid ACK(0): stop timer → go to State A<br>• Timer expires: retransmit frame(seq=0), restart timer → stay in B<br>• Receive wrong/corrupt ACK: ignore<br><b>Receiver State C (Waiting for Frame 0):</b><br>• Receive frame(0), no error: deliver to layer above → send ACK(0) → go to State D<br>• Receive frame(0) with error: discard (sender will timeout)<br>• Receive frame(1) [old duplicate]: send ACK(1) and stay in C<br><b>Receiver State D (Waiting for Frame 1): same logic with seq=1</b><br><b>Efficiency:</b> η = 1/(1+2a), where a = propagation delay/transmission delay. Terrible when a >> 1 (satellite links, high-speed WANs).
Walk through Go-Back-N — what happens when a frame is lost?	<b>Setup:</b> Window W=4, sequence numbers 0–7 (need ≥ W+1 = 5 values). Frames 0,1,2,3 in flight. Frame 2 is lost.<br><b>Sender window:</b> send_base=0, nextseqnum=4. Sends frames 0,1,2,3.<br><b>Normal reception:</b> Receiver gets frame 0 → ACK(0); frame 1 → ACK(1).<br><b>Loss of frame 2:</b><br>• Receiver gets frame 3 → out of order in GBN → discard frame 3 → send ACK(1) [last in-order]<br>• Sender receives ACK(1) — same as last ACK → no window advance<br><b>Timer expires for frame 2:</b><br>• Sender retransmits frames 2, 3 (and any others in window) — entire window from send_base<br>• Receiver now receives frame 2 → delivers 2 → ACK(2); receives frame 3 → delivers 3 → ACK(3)<br><b>Key GBN rule:</b> Receiver has NO buffer for out-of-order frames. Any frame not matching expected seq is discarded.
Walk through Selective Repeat — what happens when a frame is lost?	<b>Setup:</b> Window W=4 (≤ 2^(n-1)=4 for n=3 bits). Frames 0,1,2,3 in flight. Frame 2 is lost.<br><b>Receiver behavior:</b><br>• Gets frame 0 → deliver → ACK(0). Gets frame 1 → deliver → ACK(1).<br>• Gets frame 3 (out of order) → buffer it → send ACK(3). Gets frame 4 → buffer → ACK(4).<br><b>Sender behavior:</b><br>• Receives ACK(0), ACK(1) → advance send_base to 2; can send frames 4,5<br>• Receives ACK(3), ACK(4) → mark 3,4 as received but cannot advance (2 still missing)<br>• Timer for frame 2 expires → retransmit frame 2 only<br><b>Receiver gets frame 2:</b><br>• Delivers frames 2,3,4 in order to layer above → ACK(2)<br>• send_base advances to 5<br><b>Key SR rule:</b> Receiver buffers out-of-order frames within its window. Only missing frames are retransmitted.
Walk through CSMA/CD step by step including collision detection and BEB.	<b>Sender algorithm:</b><br>1. If channel idle: begin transmitting. If busy: wait until idle, then transmit.<br>2. While transmitting, monitor channel for collision (detected signal ≠ sent signal).<br>3. No collision until frame complete → done.<br>4. Collision detected:<br>   a. Abort transmission immediately<br>   b. Send 48-bit jam signal (ensures all stations detect the collision — remote stations may not have noticed yet)<br>   c. Increment collision count n<br>   d. If n > 16: discard frame, report error<br>   e. Choose random K from [0, 2^min(n,10) − 1]<br>   f. Wait K × 51.2 μs (one slot time for 10 Mbps Ethernet)<br>   g. Go to step 1<br><b>Minimum frame size (64 bytes):</b> Frame must be long enough that sender is still transmitting when collision signal returns. 2×propagation delay (up to 51.2 μs on 500m coax at 10 Mbps) = 512 bits = 64 bytes.
What is the Ethernet frame format — every field and its size?	<b>Preamble (7 bytes):</b> 10101010... — allows receiver to synchronize clock<br><b>SFD / Start Frame Delimiter (1 byte):</b> 10101011 — signals frame start<br><b>Destination MAC (6 bytes):</b> 48-bit address; FF:FF:FF:FF:FF:FF = broadcast; bit 0 of first byte = multicast<br><b>Source MAC (6 bytes):</b> Sender's hardware address<br><b>EtherType / Length (2 bytes):</b> ≥0x0800 → EtherType (0x0800=IPv4, 0x0806=ARP, 0x86DD=IPv6, 0x8100=802.1Q VLAN); &lt;1500 → payload length (802.3)<br><b>Payload (46–1500 bytes):</b> Layer-3 datagram; padded to 46 bytes minimum<br><b>FCS / CRC-32 (4 bytes):</b> CRC over Dest+Src+EtherType+Payload<br><b>Min frame: 64 bytes</b> (ensures CSMA/CD collision detection works). Max: 1518 bytes (1522 with VLAN tag).
How does a learning bridge build and use its forwarding table?	<b>Initial state:</b> Empty table; every frame is flooded.<br><b>On receiving frame (src=A, dst=B, port=P):</b><br>Step 1 — <b>Learn:</b> Record (A → port P, timestamp=now). If A already in table: refresh timestamp.<br>Step 2 — <b>Forward:</b><br>• Lookup B in table:<br>  - Found at port Q, Q≠P → forward out port Q only (no flooding)<br>  - Found at port Q, Q=P → drop (filter: source and dest on same segment)<br>  - Not found → flood: copy frame out all ports except P<br><b>Aging:</b> Entries not refreshed within timeout (~300 s) are deleted — handles hosts that move or disappear.<br><b>Result:</b> After a "learning period," most traffic is forwarded directly rather than flooded. The bridge is transparent to hosts — they don't know or care that it's there.
Walk through Spanning Tree Protocol (STP) step by step.	<b>Goal:</b> Eliminate loops from a bridged network while retaining redundant physical links.<br><b>Step 1 — Elect Root Bridge:</b> All bridges start believing they're root; exchange BPDUs (Bridge Protocol Data Units) with their Bridge ID (priority 16b + MAC 48b). Lowest Bridge ID = root. Root sends Hellos every 2 seconds.<br><b>Step 2 — Each non-root bridge elects Root Port:</b> Port with lowest cumulative Root Path Cost (cost to reach root). Tie-break: lowest upstream Bridge ID, then lowest upstream Port ID.<br><b>Step 3 — Each LAN segment elects Designated Port:</b> Bridge with lowest root path cost on that segment becomes DP. If tie: lowest Bridge ID. Only DP forwards on that segment.<br><b>Step 4 — Block all other ports:</b> Non-root, non-designated ports enter Blocking state.<br><b>Port states:</b> Blocking (receive BPDUs only) → Listening (15s) → Learning (15s, build MAC table) → Forwarding<br><b>Total convergence: 30–50 seconds. RSTP (802.1w) converges in ~1 second using proposal/agreement.</b>
How does 802.1Q VLAN tagging work at the frame level?	<b>A 4-byte tag is inserted between Source MAC and EtherType:</b><br><b>Tag fields:</b><br>• TPID (2 bytes): 0x8100 — identifies the frame as 802.1Q-tagged<br>• PCP (3 bits): Priority Code Point (0–7, IEEE 802.1p Class of Service)<br>• DEI (1 bit): Drop Eligible Indicator<br>• VID (12 bits): VLAN ID (1–4094; 0=priority-only; 4095=reserved)<br><b>Access port (host-facing):</b><br>• Inbound: switch accepts untagged frame, inserts tag with configured VID<br>• Outbound: switch strips tag before delivering to host<br><b>Trunk port (switch-to-switch):</b><br>• Carries tagged frames for multiple VLANs; tags are preserved end-to-end<br>• Native VLAN: untagged frames on trunk assigned to native VLAN (default 1)<br><b>Inter-VLAN routing:</b> Requires a Layer-3 hop (router or L3 switch) — VLANs are isolated at Layer 2.
How does IP fragmentation work step by step? Worked example.	<b>Trigger:</b> Datagram exceeds outgoing link's MTU.<br><b>Fragmentation algorithm:</b><br>1. Fragment payload into chunks each ≤ MTU − IP header (20 bytes)<br>2. Each fragment shares the original IP header, modified:<br>   • Identification: same for all fragments<br>   • Fragment Offset = byte_start_of_this_fragment / 8<br>   • MF (More Fragments): 1 on all except last; 0 on last<br>   • Total Length: updated per-fragment<br>3. Only the destination reassembles<br><b>Worked example:</b> 4000-byte datagram (20-byte header, 3980-byte payload), MTU=1500.<br>Max payload per fragment: 1500 − 20 = 1480 bytes<br>Frag 1: offset=0, length=1500, MF=1 (bytes 0–1479)<br>Frag 2: offset=1480/8=185, length=1500, MF=1 (bytes 1480–2959)<br>Frag 3: offset=2960/8=370, length=1060, MF=0 (bytes 2960–3979)<br><b>Reassembly:</b> Destination collects fragments with same (src IP, dst IP, protocol, ID); orders by offset; delivers when MF=0 and all gaps filled. Timer → if incomplete: discard all fragments.
How does DHCP work step by step (DORA process)?	<b>All messages use UDP: client port 68, server port 67</b><br><b>1. DISCOVER:</b> New client broadcasts DHCPDISCOVER<br>• Src IP: 0.0.0.0, Dst IP: 255.255.255.255<br>• "I need an IP address. My MAC is XX:XX:XX:XX:XX:XX"<br><b>2. OFFER:</b> Server(s) respond with DHCPOFFER<br>• Contains: offered IP, subnet mask, default gateway, DNS servers, lease time<br>• Multiple servers may respond; client picks one<br><b>3. REQUEST:</b> Client broadcasts DHCPREQUEST<br>• "I accept the offer from server Y with IP X" (broadcast so other servers know to release)<br><b>4. ACKNOWLEDGE:</b> Chosen server sends DHCPACK<br>• Confirms IP lease; client configures interface<br><b>Lease renewal:</b> At 50% of lease lifetime: unicast DHCPREQUEST to server. At 87.5%: broadcast. On expiry: must stop using IP and restart DORA.<br><b>Relay agents:</b> Routers relay DHCP broadcasts between subnets by unicasting to a configured DHCP server IP.
How does ARP work step by step?	<b>Goal:</b> Host A needs the MAC address for IP 192.168.1.5 (same subnet).<br>1. Check ARP cache → if entry found and not expired: use cached MAC. Done.<br>2. Cache miss → A broadcasts ARP Request:<br>   • Ethernet dst=FF:FF:FF:FF:FF:FF, src=A's MAC<br>   • ARP: "Who has 192.168.1.5? Tell 192.168.1.1"<br>   • Sender MAC=A's MAC, Sender IP=A's IP, Target MAC=00:00:00:00:00:00, Target IP=192.168.1.5<br>3. All hosts receive the broadcast; only 192.168.1.5 responds:<br>   • Unicast ARP Reply: "192.168.1.5 is at B's MAC"<br>4. A stores (192.168.1.5 → B's MAC) in ARP cache (TTL ~20 min)<br>5. A sends IP packet wrapped in Ethernet frame with dst=B's MAC<br><b>For remote destinations:</b> A ARPs for the default gateway's MAC (not the remote host's IP). Gateway routes the IP packet onward.<br><b>Gratuitous ARP:</b> Host announces own IP→MAC at startup — used for duplicate IP detection and failover.
How does CIDR route aggregation work? Show a worked example.	<b>Motivation:</b> Before CIDR, each network needed a separate routing table entry. CIDR allows multiple networks to be summarized into one.<br><b>Aggregation example:</b><br>ISP has: 200.23.16.0/24, 200.23.17.0/24, 200.23.18.0/24, 200.23.19.0/24<br>All share first 22 bits → advertise as one route: 200.23.16.0/22 (covers all four /24s)<br>Reduces 4 routing table entries to 1.<br><b>Longest prefix match rule:</b> If a more-specific route also exists in the table, it takes priority over the aggregate.<br><b>Example:</b> Table has 200.23.16.0/22 (aggregate) AND 200.23.18.0/24 (specific).<br>Packet to 200.23.18.5 matches both → /24 wins (longest prefix).<br>Packet to 200.23.17.5 matches only /22 → uses aggregate.<br><b>Patricia trie (radix tree):</b> Software-based longest-prefix-match. Walk bit by bit from MSB; track last matching prefix; return it at end.
How does NAT work step by step? Give a worked example.	<b>Setup:</b> Private network 10.0.0.0/24 behind NAT router with public IP 203.0.113.1.<br><b>Outbound (client 10.0.0.5 port 3000 → 8.8.8.8 port 53):</b><br>1. Packet arrives at NAT router: src=10.0.0.5:3000, dst=8.8.8.8:53<br>2. NAT allocates a public port (e.g., 5000)<br>3. Records in NAT table: (10.0.0.5:3000 ↔ 203.0.113.1:5000, dst=8.8.8.8:53)<br>4. Rewrites packet: src=203.0.113.1:5000, dst=8.8.8.8:53<br>5. Forwards to Internet<br><b>Inbound (response: 8.8.8.8:53 → 203.0.113.1:5000):</b><br>6. NAT receives packet with dst=203.0.113.1:5000<br>7. Looks up port 5000 in NAT table → finds (10.0.0.5:3000)<br>8. Rewrites: dst=10.0.0.5:3000<br>9. Forwards to internal host<br><b>Drawbacks:</b> Breaks end-to-end principle; breaks protocols with IP in payload (SIP, FTP); inbound connections require manual port forwarding; P2P requires STUN/TURN/ICE.
Trace through Dijkstra's algorithm on a network graph.	<b>Algorithm:</b> Finds shortest paths from one source to all destinations.<br><b>Data structures:</b> dist[] (best known distance), visited set, priority queue (min-heap on dist)<br><b>Init:</b> dist[src]=0, dist[all others]=∞<br><b>Each iteration:</b><br>1. Extract node u with minimum dist[u] from priority queue<br>2. Mark u visited (never revisit)<br>3. For each unvisited neighbor v of u: if dist[u]+cost(u,v) < dist[v]: update dist[v], set predecessor[v]=u<br><b>Example: A-B=2, A-C=4, B-C=1, B-D=7, C-D=3. Source=A:</b><br>Init: A=0,B=∞,C=∞,D=∞<br>Round 1: Extract A(0). Update B=2(via A), C=4(via A).<br>Round 2: Extract B(2). Update C=min(4,3)=3(via B), D=min(∞,9)=9(via B).<br>Round 3: Extract C(3). Update D=min(9,6)=6(via C).<br>Round 4: Extract D(6). No updates.<br><b>Result:</b> A→B=2, A→B→C=3, A→B→C→D=6<br><b>Complexity:</b> O((V+E)logV) with binary heap.
Trace through the Distance Vector algorithm — initialization and convergence.	<b>Bellman-Ford equation:</b> D(x,y) = min_v { cost(x,v) + D(v,y) }<br><b>Example: 3 nodes. A-B cost=1, B-C cost=2, A-C cost=∞ initially.</b><br><b>Initialization (each node knows only its neighbors):</b><br>A: D(A,A)=0, D(A,B)=1, D(A,C)=∞<br>B: D(B,A)=1, D(B,B)=0, D(B,C)=2<br>C: D(C,A)=∞, D(C,B)=2, D(C,C)=0<br><b>Round 1 — A receives B's vector:</b><br>D(A,C) = min(∞, cost(A,B)+D(B,C)) = min(∞, 1+2) = 3<br>A's updated table: A=0, B=1, C=3 via B<br><b>Round 2 — No further improvements → converged.</b><br><b>Trigger updates:</b> When a link cost changes or fails, immediately send updated vector to neighbors. Don't wait for the periodic timer (RIP: 30s).<br><b>Weakness:</b> Count-to-infinity problem on link failures.
Walk through the count-to-infinity problem with a specific example.	<b>Topology:</b> A—B—C. Costs A-B=1, B-C=1. Converged: A knows C at cost 2 (via B); B knows C at cost 1 (direct).<br><b>B-C link breaks:</b><br>B sets D(B,C)=∞. B checks neighbors: D(B,C) via A = cost(B,A)+D(A,C) = 1+2=3. B updates: D(B,C)=3 via A.<br>B advertises to A: "I reach C at cost 3."<br>A updates: D(A,C)=1+3=4 via B. Advertises to B: "I reach C at cost 4."<br>B updates: D(B,C)=1+4=5 via A.<br>...continues until cost reaches 16 (RIP's infinity).<br><b>Root cause:</b> A's path to C was through B — when B's direct link fails, A's stale route through B appears as a valid alternative to B. Phantom route.<br><b>Split horizon fix:</b> B never advertises C back to A (since B learned C from A). A's vector is invisible to B for that route → when B's direct link fails, B correctly sees D(B,C)=∞ with no alternative.<br><b>Poison reverse:</b> B explicitly tells A "D(B,C)=∞" — stronger fix.
Walk through BGP route selection step by step — all 9 tie-breaking criteria.	When multiple BGP paths exist to the same prefix, choose in order:<br>1. <b>Weight (Cisco-specific, local only):</b> Highest preferred. Not in BGP standard.<br>2. <b>LOCAL-PREF:</b> Highest preferred. Set per-AS; controls which exit from the AS to prefer (outbound policy).<br>3. <b>Locally originated:</b> Routes this router originated (via network/redistribute) preferred over received routes.<br>4. <b>AS-PATH length:</b> Shortest preferred. Fewer ASes in path = closer destination.<br>5. <b>ORIGIN:</b> IGP (i) < EGP (e) < Incomplete (?). Prefer routes learned via IGP.<br>6. <b>MED (lower preferred):</b> Only compared among routes from the same neighboring AS. Suggests preferred entry point.<br>7. <b>eBGP > iBGP:</b> Externally learned routes preferred over internally redistributed.<br>8. <b>IGP cost to NEXT-HOP (lowest):</b> "Hot potato" — exit AS via the closest exit point.<br>9. <b>Router-ID (lowest):</b> Final arbitrary tiebreaker.<br><b>First criterion producing a unique winner stops the process.</b>
What does a BGP UPDATE message contain and how does it propagate?	<b>BGP UPDATE message fields:</b><br>• <b>Withdrawn Routes:</b> List of prefixes no longer reachable (withdrawals sent before new announcements)<br>• <b>Path Attributes:</b> AS-PATH, NEXT-HOP, LOCAL-PREF, MED, ORIGIN, COMMUNITY, AGGREGATOR, etc.<br>• <b>NLRI (Network Layer Reachability Information):</b> List of IP prefixes reachable via these attributes<br><b>Propagation:</b><br>• eBGP route received → advertise to iBGP peers + other eBGP peers (prepend own AS to AS-PATH)<br>• iBGP route received → advertise to eBGP peers only (NOT to other iBGP peers — loop prevention)<br><b>NEXT-HOP:</b><br>• To eBGP peer: set NEXT-HOP = own interface IP<br>• To iBGP peers: preserve original eBGP NEXT-HOP (internal routers use IGP to reach it)<br><b>TCP keepalive:</b> BGP sessions run over TCP 179; KEEPALIVE every 60s; HOLD timer 180s before session declared dead.
Walk through OSPF neighbor formation and LSDB synchronization.	<b>OSPF neighbor state machine (8 states):</b><br><b>Down → Init:</b> Hello received from neighbor; our Router-ID not yet in neighbor's hello list<br><b>Init → 2-Way:</b> Our Router-ID appears in neighbor's hello → bidirectional confirmed. Walk through the full TCP connection lifecycle with actual sequence numbers.	<b>Setup (3-way handshake):</b><br>1. Client: SYN, Seq=100 (Initial Sequence Number (ISN) chosen randomly)<br>2. Server: SYN+ACK, Seq=300 (server's Initial Sequence Number (ISN)), Ack=101 (expects byte 101 next)<br>3. Client: ACK, Seq=101, Ack=301 → ESTABLISHED<br><b>Data: Client sends "Hello" (5 bytes):</b><br>4. Client: Seq=101, Ack=301, Data="Hello" (5 bytes)<br>5. Server: ACK=106 (next expected = 101+5=106)<br><b>Server replies "Hi" (2 bytes):</b><br>6. Server: Seq=301, Ack=106, Data="Hi"<br>7. Client: ACK=303<br><b>Teardown:</b><br>8. Client: FIN, Seq=106 → FIN_WAIT_1<br>9. Server: ACK=107 → Client enters FIN_WAIT_2; Server enters CLOSE_WAIT<br>10. Server: FIN, Seq=303 → LAST_ACK<br>11. Client: ACK=304 → Client enters TIME_WAIT; Server enters CLOSED<br>After 2×MSL: Client enters CLOSED.
What are all the TCP states and their transitions?	<b>CLOSED:</b> No connection. send SYN → SYN_SENT; receive SYN → SYN_RCVD (passive open via LISTEN).<br><b>LISTEN:</b> Passive open. Receive SYN → send SYN-ACK → SYN_RCVD.<br><b>SYN_SENT:</b> SYN sent. Receive SYN-ACK → send ACK → ESTABLISHED. Timeout → CLOSED.<br><b>SYN_RCVD:</b> Receive ACK → ESTABLISHED. Receive RST → LISTEN.<br><b>ESTABLISHED:</b> Data flows. Local close: send FIN → FIN_WAIT_1. Receive FIN → send ACK → CLOSE_WAIT.<br><b>FIN_WAIT_1:</b> Sent FIN. Receive ACK → FIN_WAIT_2. Receive FIN+ACK → TIME_WAIT.<br><b>FIN_WAIT_2:</b> Our FIN ACKed. Receive FIN → send ACK → TIME_WAIT.<br><b>CLOSE_WAIT:</b> Received peer's FIN; app still open. App closes → send FIN → LAST_ACK.<br><b>LAST_ACK:</b> Receive ACK → CLOSED.<br><b>TIME_WAIT:</b> Wait 2×MSL (~120s). Timer expires → CLOSED.<br><b>CLOSING:</b> Simultaneous close; both sent FIN. Receive ACK → TIME_WAIT.<br><b>TIME_WAIT purpose:</b> Absorb delayed segments; retransmit final ACK if needed.
Trace TCP congestion control through a full sawtooth (connection start → loss → recovery).	<b>Phase 1 — Slow Start (SS), ssthresh=64:</b><br>cwnd: 1→2→4→8→16→32→64 Maximum Segment Size (MSS) (doubles each Round-Trip Time (RTT) via Acknowledgment (ACK)-by-ACK increase)<br><b>Phase 2 — Congestion Avoidance (CA), cwnd=64:</b><br>cwnd: 64→65→66→...→88 MSS (linear, +1 MSS/RTT)<br><b>Loss detected: Triple dup ACK at cwnd=88 (TCP Reno):</b><br>ssthresh = 88/2 = 44<br>cwnd = 44 → enter Fast Recovery; retransmit lost segment<br>Each additional dup ACK: cwnd++<br>New ACK → cwnd = 44 → enter CA<br><b>CA continues linearly from cwnd=44:</b><br>44→45→46→...<br><b>Timeout loss at cwnd=X:</b><br>ssthresh = X/2; cwnd = 1 MSS<br>Restart SS from 1 → exponential growth until ssthresh → switch to CA.<br><b>TCP Tahoe vs Reno:</b> Tahoe always resets cwnd=1 on any loss. Reno only resets on timeout; uses Fast Recovery on dup ACKs.
Walk through TCP Fast Retransmit and Fast Recovery step by step.	<b>Scenario:</b> Sender has cwnd=20 Maximum Segment Size (MSS), ssthresh=32. Segment 8 is lost. Segments 9,10,11 arrive at receiver.<br><b>Receiver side:</b><br>• Expects segment 8; receives 9 → send dup ACK(7) [last in-order]<br>• Receives 10 → send dup ACK(7)<br>• Receives 11 → send dup ACK(7)<br><b>Sender side (3rd dup ACK received):</b><br>1. Fast Retransmit: immediately resend segment 8 (don't wait for timeout)<br>2. ssthresh = cwnd/2 = 20/2 = 10<br>3. cwnd = ssthresh + 3 = 10+3 = 13 (inflate for 3 segments that left network)<br>4. Enter Fast Recovery<br><b>During Fast Recovery (if more dup ACKs arrive):</b><br>• Each additional dup ACK: cwnd += 1 (allows sending new segments beyond the lost one)<br><b>New (non-duplicate) ACK arrives (ACKs segment 8 and beyond):</b><br>5. Deflate: cwnd = ssthresh = 10<br>6. Exit Fast Recovery; enter Congestion Avoidance (+1 MSS/RTT).
How does the Jacobson/Karels adaptive RTO work step by step?	<b>Goal:</b> Compute a timeout that adapts to actual RTT and its variance to avoid spurious retransmits and slow recovery.<br><b>On receiving each non-retransmitted ACK:</b><br><b>Step 1 — Measure SampleRTT:</b> time_of_ACK − time_segment_was_sent<br><b>Step 2 — Update SRTT (smoothed RTT):</b><br>Diff = SampleRTT − SRTT<br>SRTT = SRTT + (0.125 × Diff) [EWMA, α=1/8]<br><b>Step 3 — Update DevRTT (variance estimate):</b><br>DevRTT = DevRTT + 0.25 × (|Diff| − DevRTT) [EWMA, β=1/4]<br><b>Step 4 — Compute RTO:</b><br>RTO = SRTT + 4 × DevRTT<br><b>Why 4×DevRTT?</b> Keeps RTO comfortably above SRTT even in jittery networks (4σ if Gaussian), preventing spurious retransmits.<br><b>Karn/Partridge rule:</b> Never sample RTT for retransmitted segments (can't distinguish which transmission the ACK acknowledges).<br><b>On timeout → Binary Exponential Backoff:</b> RTO = 2×RTO (cap at 60s). Reset to computed value after successful non-retransmit ACK.
How does TCP flow control with rwnd work, including the zero window scenario?	<b>rwnd (receiver window):</b> In every ACK, the receiver advertises how many bytes it can accept into its buffer right now.<br><b>Sender constraint:</b> LastByteSent − LastByteAcked ≤ rwnd<br>Effective sending rate is limited to min(cwnd, rwnd).<br><b>Normal flow:</b><br>App writes data slowly → receiver buffer mostly empty → rwnd stays large → sender unconstrained.<br>App reads slowly → receiver buffer fills → rwnd decreases → sender slows down.<br><b>Zero window scenario:</b><br>1. Receiver buffer fills → ACK with rwnd=0 → sender stops sending<br>2. Sender starts zero-window probe timer (starts at Retransmission Timeout (RTO), doubles like Binary Exponential Backoff (BEB))<br>3. Sender sends 1-byte probe segments periodically<br>4. Receiver ACKs the probe with current rwnd (even if still 0)<br>5. When buffer clears → receiver ACKs with rwnd>0 → sender resumes<br><b>Silly Window Syndrome (SWS):</b> Receiver advertises tiny windows (1 byte) → tiny segments → huge overhead. Fixed by Clark's Algorithm: receiver won't advertise window until it can accept min(Maximum Segment Size (MSS), half buffer size).
Walk through QUIC handshake in 0-RTT and 1-RTT mode.	<b>Problem with TLS over TCP:</b> TCP 3-way handshake (1 RTT) + TLS 1.3 (1 RTT) = 2 RTTs before any application data.<br><b>QUIC 1-RTT (first connection to this server):</b><br>→ Client sends Initial (QUIC header + TLS ClientHello, key_share)<br>← Server sends Initial + Handshake + 1-RTT packets: ServerHello, cert, CertVerify, Finished (all in one flight)<br>→ Client sends Finished + first encrypted HTTP/3 request (1 RTT to start data)<br>Total: <b>1 RTT handshake</b><br><b>QUIC 0-RTT (returning client with cached session ticket):</b><br>→ Client sends Initial + 0-RTT data encrypted with cached key (HTTP request in very first UDP packet)<br>← Server processes 0-RTT data immediately; responds with new session keys<br>Total: <b>0 RTTs to first byte of data</b><br><b>0-RTT limitation:</b> Replay-unsafe — attacker could replay the 0-RTT packet. Restrict to idempotent requests (GET, not POST).<br><b>Connection ID:</b> QUIC identifies connections by a Connection ID, not the IP:port 4-tuple → survives NAT rebinding and WiFi→LTE handoffs.

Walk through HTTP/2 stream multiplexing with a frame-level trace.	<b>Client requests HTML (Stream 1) and CSS (Stream 3) over one TCP connection:</b><br>[→] HEADERS frame | Stream 1 | GET /index.html HTTP/1.1<br>[→] HEADERS frame | Stream 3 | GET /style.css HTTP/1.1<br>[←] HEADERS frame | Stream 3 | 200 OK (CSS ready first — server sends out-of-order!)<br>[←] DATA frame   | Stream 3 | "body { color:red; }" (Stream 3 complete)<br>[←] HEADERS frame | Stream 1 | 200 OK<br>[←] DATA frame   | Stream 1 | "&lt;html&gt;...first chunk"<br>[←] DATA frame   | Stream 1 | "...&lt;/html&gt;" (Stream 1 complete)<br><b>HTTP/1.1 with pipelining would have blocked CSS until HTML was done.</b><br><b>HPACK:</b> First request sends full headers. Second request sends only the delta (e.g., just the path changes; Host, Accept, Cookie sent as 1-byte index references). Saves 80–90% of header bytes on average.<br><b>Transport-layer HOL (TCP):</b> If a TCP segment carrying a Stream 1 DATA frame is lost, the TCP layer stalls delivery of ALL data — including already-arrived Stream 3 frames. HTTP/3+QUIC fixes this.
How does HTTP/3 over QUIC eliminate transport-layer HOL blocking?	<b>Root cause:</b> TCP delivers data as one byte stream. One lost TCP segment stalls all data in the receive buffer — OS cannot pass out-of-order bytes to the application. This stalls ALL HTTP/2 streams, even those whose data arrived intact.<br><b>QUIC solution:</b> Each QUIC stream has independent flow control and ordering. Reliability is stream-scoped.<br><b>How it works on a loss:</b><br>1. UDP packet carrying QUIC Stream 1 data is lost (gap in packet numbers detected)<br>2. QUIC marks Stream 1 as blocked pending retransmit<br>3. Meanwhile, the OS receives subsequent UDP packets (for Stream 3) — QUIC passes Stream 3's complete data to the browser immediately<br>4. Only Stream 1 waits for its retransmitted packet<br>5. Browser renders Stream 3 (CSS) and renders what it can of Stream 1 while waiting<br><b>Summary:</b> TCP's byte-stream abstraction means all-or-nothing delivery per connection. QUIC's per-stream abstraction means independent delivery per stream.
How does CDN DNS-based redirection work step by step?	<b>Scenario:</b> User in Seattle accesses www.example.com hosted by a CDN with PoPs in Seattle, Denver, NYC.<br>1. How does BGP prevent routing loops?	<b>Mechanism: AS-PATH attribute loop detection.</b><br>Every BGP UPDATE carries the full sequence of AS numbers the route has traversed.<br><b>Rule:</b> When a BGP speaker receives an UPDATE, if its own ASN appears anywhere in the AS-PATH → silently discard the route.<br><b>Step-by-step example:</b><br>AS1 originates 10.0.0.0/8 → sends UPDATE to AS2 with AS-PATH=[1]<br>AS2 prepends itself → sends to AS3 with AS-PATH=[2,1]<br>AS3 prepends itself → sends to AS1 with AS-PATH=[3,2,1]<br>AS1 sees its own ASN (1) in [3,2,1] → discards → no loop<br><b>iBGP loop prevention:</b> Routes learned from an iBGP peer are never re-advertised to other iBGP peers. This requires full-mesh iBGP or Route Reflectors.<br><b>NEXT-HOP:</b> eBGP peer sets NEXT-HOP to its own address. Core routers use IGP to reach the NEXT-HOP, so BGP routes are always "anchored" to reachable next-hops.
Walk through SIP call setup with a complete message trace.	<b>Alice (sip:alice@atlanta.com) calls Bob (sip:bob@biloxi.com):</b><br>1. After ACK, proxy is out of the picture — media never touches it.
Walk through OSPF LSA flooding and duplicate suppression.	<b>LSA originates when local state changes (link up/down, neighbor gained/lost, cost change):</b><br>1. Router generates LSA with: Router-ID, LS-Sequence-Number (incremented), LS-Age=0, Link list + costs<br>2. Floods LSA out ALL interfaces except the one it arrived on<br>3. Each receiving router:<br>   a. Check LS-Sequence-Number: newer than stored? → update LSDB and re-flood<br>   b. Same or older? → discard (already processed this LSA)<br>   c. Send LS-ACK back to sender (reliable flooding)<br>4. Retransmit if no ACK received within timeout (per-neighbor retransmit timer)<br><b>On broadcast networks (with DR):</b><br>• Non-DR routers send LSAs to 224.0.0.6 (AllDRRouters)<br>• DR re-floods to 224.0.0.5 (AllSPFRouters)<br>• Reduces messages from O(n²) to O(n)<br><b>LSA aging:</b> Refreshed every 30 minutes (new sequence number). LS-Age increments as LSA sits in LSDB. At MaxAge (3600s) → LSA is flushed and a new flood is initiated if node still reachable.
How does RTP/RTCP remove jitter and provide QoS feedback?	<b>Sender (G.711 VoIP, 8kHz, 20ms packets):</b><br>1. Sample audio at 8 kHz → 160 samples per 20ms packet<br>2. Wrap in RTP header: Seq# (increments by 1 per packet), Timestamp (increments by 160 per packet = samples), SSRC (stream ID), Payload Type=0 (G.711)<br>3. Send over UDP; network introduces variable delay (jitter)<br><b>Receiver — jitter removal via playout buffer:</b><br>1. Set playout delay = 50ms (absorbs most jitter)<br>2. Packet arrives at t=0ms real clock → scheduled to play at virtual time T+50ms<br>3. Packet 2 arrives at t=22ms → play at T+70ms (irregular arrival, regular playback)<br>4. Packet 3 arrives at t=19ms → play at T+90ms<br>5. If packet arrives after its scheduled play time → lost/concealed (repeat last frame, generate comfort noise)<br><b>RTCP RR (Receiver Report, sent every ~5s):</b><br>Reports to sender: fraction lost (last interval), cumulative lost, highest seq# received, inter-arrival jitter (samples), timestamp of last SR + delay since last SR<br><b>Sender adapts:</b> High loss reported → reduce bitrate or switch to lower-quality codec.
How does Fair Queuing compute virtual finish times? Worked example.	<b>Goal:</b> Approximate bit-by-bit round-robin (perfectly fair) using packet-sized chunks.<br><b>Virtual clock:</b> Advances 1 tick per bit transmitted across all flows.<br><b>Finish time formula per packet:</b><br>F(j, F) = max(Arrive(j, F), F(j-1, F)) + Length(j, F)<br>• Arrive = virtual clock time when packet j of flow F arrives<br>• F(j-1, F) = finish time of previous packet from same flow (or 0 if first)<br>• Length = packet length in bits<br><b>Service order:</b> Serve packets in ascending order of virtual finish time.<br><b>Example: 2 flows, virtual time starts at 0.</b><br>Flow A sends 100-bit packet at vtime=0: F(1,A) = max(0,0) + 100 = 100<br>Flow B sends 200-bit packet at vtime=50: F(1,B) = max(50,0) + 200 = 250<br>Flow A sends 100-bit packet at vtime=200: F(2,A) = max(200,100) + 100 = 300<br>Service order: A-pkt1 (100), B-pkt1 (250), A-pkt2 (300)<br><b>WFQ with weight W_F:</b> F(j,F) = max(Arrive, F(j-1,F)) + Length/W_F → higher weight → smaller increment → served sooner → more bandwidth.
How does MPLS label switching work step by step through a network?	<b>LSP setup (via LDP or RSVP-TE):</b> Path is pre-signaled; each router along the path knows what label to expect and what label to output.<br><b>Ingress LER (Label Edge Router) — PUSH operation:</b><br>1. Receives plain IP packet<br>2. Classifies into FEC (Forwarding Equivalence Class) based on dst prefix, DSCP, or VPN<br>3. Pushes MPLS label (e.g., label=100) between L2 header and IP header<br>4. Forwards out pre-determined interface<br><b>Transit LSR (Label Switching Router) — SWAP operation:</b><br>1. Reads incoming label=100<br>2. Looks up label=100 in LFIB (Label Forwarding Information Base)<br>3. Swaps label 100 → label 200; forwards out pre-determined interface<br>4. No IP routing lookup needed — O(1) exact-match vs. O(32) LPM<br><b>Egress LER — POP operation (or PHP — Penultimate Hop Popping):</b><br>1. Receives labeled packet<br>2. Pops label stack (or receives already-popped from penultimate hop)<br>3. Delivers plain IP packet to destination subnet<br><b>PHP benefit:</b> Egress LER does only IP lookup, not both label lookup + IP lookup → reduces processing at the last hop.
How does Mobile IP handle packet delivery and what is the Triangle Routing Problem?	<b>Mobile host roaming to a foreign network:</b><br>1. Mobile host connects to foreign network<br>2. Gets Care-of Address (CoA) — temp IP on foreign network — from Foreign Agent (FA) or DHCP<br>3. Mobile host registers CoA with Home Agent (HA) at home network (sends registration request)<br><b>Incoming packet flow:</b><br>4. Correspondent sends packet to Mobile's Home Address (doesn't know mobile moved)<br>5. HA intercepts packet (via proxy ARP — HA answers ARP for mobile's home address)<br>6. HA encapsulates (IP-in-IP tunnel) original packet in new IP header: src=HA, dst=CoA<br>7. FA receives tunneled packet, decapsulates, delivers to mobile<br><b>Triangle Routing Problem:</b> Even if correspondent and mobile are co-located in Tokyo, traffic goes: Tokyo → New York (Home Network) → Tokyo. Doubled latency, wasted bandwidth.<br><b>Route Optimization fix:</b> HA sends Binding Update to correspondent. Correspondent caches CoA → tunnels directly to mobile. Eliminates the triangle — but requires correspondent to support Mobile IP extensions.
How does OSPF handle equal-cost paths (ECMP) and what determines link cost?	<b>Default OSPF link cost:</b> 10^8 / interface_bandwidth_bps<br>FastEthernet (100 Mbps): 10^8/10^8 = 1<br>GigabitEthernet (1 Gbps): 10^8/10^9 = 0.1 → rounded to 1 (same as FastEthernet!)<br>Serial (1.544 Mbps T1): 10^8/1544000 ≈ 64<br><b>Admin should tune costs manually for Gigabit+ links</b> to distinguish them.<br><b>ECMP (Equal-Cost Multi-Path):</b><br>When Dijkstra finds multiple paths with identical total cost:<br>1. All equal-cost paths are installed in the forwarding table<br>2. Traffic distributed across them (load balancing)<br>3. <b>Per-packet (round-robin):</b> Maximizes bandwidth utilization but can reorder TCP segments<br>4. <b>Per-flow (5-tuple hash):</b> All packets of the same TCP flow take the same path — preserves ordering. Preferred in practice.<br><b>Summarization at ABRs:</b> Multiple specific prefixes within an area can be summarized into one range for advertisement to other areas → reduces LSA flooding scope and routing table size.
How does Nagle's Algorithm work and when should it be disabled?	<b>Problem:</b> Apps sending 1-byte writes (interactive SSH keystrokes) generate 1-byte payload segments with 40 bytes of TCP+IP header = 1% efficiency.<br><b>Nagle's rule:</b> A sender MAY transmit a new small segment only if either:<br>1. No previously sent data is currently outstanding (pipe is empty), OR<br>2. The buffered data equals or exceeds one full Maximum Segment Size (MSS)<br>Otherwise: hold small data in the send buffer and wait for ACK to clear the pipe.<br><b>Effect:</b> Small writes are coalesced into one larger segment once the ACK arrives, improving efficiency dramatically.<br><b>Delayed ACK interaction problem:</b> Receiver delays ACK up to 200ms hoping to piggyback. If sender is waiting for ACK (Nagle), and receiver is waiting to ACK (delayed ACK), neither sends → 200ms deadlock. Fix: disable Nagle via TCP_NODELAY socket option.<br><b>When to disable Nagle (TCP_NODELAY):</b> Latency-sensitive apps — SSH, Telnet, online games, algorithmic trading, mouse/keyboard events in RDP.
How does the TCP pseudo-header protect against misdelivery?	<b>The TCP checksum covers three things:</b><br>1. TCP header (20+ bytes)<br>2. TCP payload (data)<br>3. An IP pseudo-header (computed at sender/receiver, NOT transmitted)<br><b>Pseudo-header fields (12 bytes):</b><br>• Source IP (32 bits)<br>• Destination IP (32 bits)<br>• Zero (8 bits) padding<br>• Protocol field (8 bits, = 6 for TCP)<br>• TCP Segment Length (16 bits, header+data)<br><b>Why include IP addresses in the checksum?</b><br>Without it: a TCP segment delivered to the wrong host OR wrong protocol could still pass the TCP checksum (since IP header is not covered by TCP's own checksum). The pseudo-header detects:<br>• Router misrouting (wrong destination IP)<br>• Demultiplexing errors (wrong protocol field)<br>• Corruption of IP layer addresses<br><b>Verification:</b> Receiver reconstructs pseudo-header from its own IP layer; adds it into the 1's complement sum. If result ≠ all-1s → packet corrupted or misrouted.
How does BGP Route Reflection solve the iBGP full-mesh scaling problem?	<b>Problem:</b> iBGP cannot re-advertise routes between iBGP peers (loop prevention). Every router needs a direct iBGP session with every other router → O(n²) sessions. With 100 iBGP routers: 4950 sessions.<br><b>Route Reflector (RR) solution:</b><br>One designated router (the RR) is allowed to re-advertise iBGP routes to its clients.<br><b>RR reflection rules:</b><br>• Route learned from eBGP → advertise to all clients AND all non-client iBGP peers<br>• Route learned from an iBGP client → advertise to all other clients AND all non-client iBGP peers<br>• Route learned from non-client iBGP peer → advertise to clients only (not to other non-clients)<br><b>Loop prevention:</b> ORIGINATOR-ID (Router-ID of route originator) + CLUSTER-LIST (list of RR cluster IDs traversed). If RR sees its own Cluster-ID in CLUSTER-LIST → discard.<br><b>Deployment:</b> Clients connect only to their RR(s) — reduces sessions from O(n²) to O(n). Use 2+ RRs per cluster for redundancy.
How does IPv6 SLAAC work step by step?	<b>Goal:</b> Hosts configure their own global IPv6 address without a DHCPv6 server.<br>1. <b>Generate link-local address:</b> fe80::/10 prefix + 64-bit interface ID (EUI-64 from MAC: insert FF:FE between bytes 3 and 4, flip 7th bit; or use random for privacy)<br>2. <b>DAD (Duplicate Address Detection):</b> Send ICMPv6 Neighbor Solicitation (NS) for own tentative address to its Solicited-Node multicast address. Wait 1 second. If Neighbor Advertisement (NA) received → collision → use another address.<br>3. <b>Router Solicitation:</b> Host multicasts RS (ICMPv6 Type 133) to ff02::2 (All-Routers)<br>4. <b>Router Advertisement (RA):</b> Router responds with RA (ICMPv6 Type 134) containing:<br>   • Network prefix(es) + prefix length (usually /64)<br>   • A flag: use SLAAC for address<br>   • M flag: use DHCPv6 for address (if set)<br>   • O flag: use DHCPv6 for other config (DNS, NTP)<br>   • Router lifetime, MTU<br>5. <b>Form global address:</b> RA prefix (64 bits) + interface ID (64 bits) = 128-bit global address<br>6. <b>DAD on global address:</b> Same as step 2<br>7. Host now has link-local + global addresses; uses router as default gateway.
How does SNMP polling and trap delivery work mechanically?	<b>Architecture:</b> NMS (Network Management Station) ↔ Agents on managed devices.<br><b>GET flow (NMS polls agent):</b><br>1. NMS sends GetRequest UDP to agent port 161<br>2. Payload: community string ("public"/"private"), list of OIDs to read<br>3. Agent looks up each OID in its MIB and returns current value<br>4. Agent replies: GetResponse with OID=value pairs<br>5. NMS stores values, graphs metrics, evaluates thresholds<br><b>TRAP flow (event-driven, agent pushes to NMS):</b><br>1. Event occurs on device (interface down, CPU > 95%, power failure)<br>2. Agent immediately sends TRAP UDP packet to NMS port 162<br>3. TRAP payload: enterprise OID (event type), timestamp, variable bindings (context data)<br>4. NMS correlates trap with device, triggers alert/ticket<br><b>Problem:</b> UDP TRAPs are fire-and-forget. SNMPv2 INFORMs add acknowledged delivery.<br><b>SNMPv3 security:</b> Adds MD5/SHA message authentication (prevents forgery) + DES/AES encryption (prevents sniffing of community strings). SNMPv1/v2c send community strings in plaintext.
How does the RSVP resource reservation process work step by step?	<b>RSVP (Resource Reservation Protocol) — IntServ signaling:</b><br><b>Step 1 — PATH message (sender → receiver along data path):</b><br>Sender sends PATH message to receiver's address. Routed by IP like normal data.<br>Each router on path:<br>• Records "path state": previous hop, PHOP (previous hop address), Tspec (sender's traffic spec: rate, burst, peak)<br>• Stores this state (soft state — must be refreshed)<br>• Forwards PATH toward receiver<br><b>Step 2 — RESV message (receiver → sender, following recorded path):</b><br>Receiver decides what resources it wants, sends RESV toward sender.<br>Each router on path:<br>• Performs admission control: "Can I grant this reservation given my current load?"<br>• If YES: reserve resources (configure scheduler queue for this flow)<br>• If NO: return RESV-ERROR to receiver — reservation rejected<br>• Forward RESV toward sender<br><b>Step 3 — Data flows with guaranteed QoS.</b><br><b>Step 4 — Maintenance:</b> PATH and RESV must be refreshed every 30 seconds (soft state). No refresh → resources auto-released. Handles router restarts cleanly.<br><b>Why it failed at scale:</b> Every router holds per-flow state. Backbone = millions of flows = infeasible.
How does consistent hashing work in a DHT and why does it matter for node joins/leaves?	<b>Basic hashing problem:</b> If you hash objects to servers using key % N, adding or removing a server remaps almost ALL keys → massive redistribution.<br><b>Consistent hashing solution:</b><br>1. Place both nodes and keys on a circular ID ring (e.g., 0 to 2^128 - 1 for Chord)<br>2. Each node manages the keys between it and its predecessor on the ring<br>3. An object is stored at the node with the smallest ID ≥ object's hash ID ("successor node")<br><b>Adding a node X between N1 and N2:</b><br>• X takes over keys that hashed between N1 and X from N2<br>• Only O(K/N) keys move (K=total keys, N=nodes) — minimal disruption<br><b>Removing a node N1:</b><br>• N1's keys transfer to N1's successor<br>• Only those keys are affected<br><b>Chord lookup:</b><br>• Each node maintains a "finger table" with O(log N) entries pointing to nodes at exponentially increasing ring distances<br>• Lookup for key K: forward to finger closest to K without overshooting → O(log N) hops
How does OSPF route summarization and area design reduce network overhead?	<b>Problem:</b> Without areas, every OSPF router floods LSAs to every other router in the AS. For 500 routers: 500 LSAs flooded 500 times = 250,000 LSA transmissions just for initial sync. Dijkstra runs on a 500-node graph.<br><b>Area design rules:</b><br>• All non-backbone areas must connect to Area 0 (backbone) directly or via virtual link<br>• ABRs (Area Border Routers) maintain separate LSDB for each connected area<br><b>How summarization reduces overhead:</b><br>1. Within Area 1: 20 routers, each with 3 links = 60 LSAs flooded within Area 1<br>2. ABR summarizes Area 1 into: 192.168.1.0/24 (instead of 20 Type-1 LSAs per router)<br>3. Area 0 only sees 1 Type-3 Summary-LSA from the ABR (not 60 LSAs)<br>4. Routers in Area 2 only know 192.168.1.0/24 exists; they don't know the internal topology of Area 1<br><b>Benefit:</b> Dijkstra in Area 2 runs on Area-2 topology only (small). Total LSDB per router = local area + summaries from other areas (not all routers everywhere).<br><b>Tradeoff:</b> Summarization can cause suboptimal routing (specific paths hidden behind summaries).
What is the end-to-end principle and how does it justify TCP's design?	<b>End-to-End Argument (Saltzer, Reed, Clark 1984):</b><br>"Functions placed in the core network are redundant if they can only be correctly implemented at the endpoints using application-level knowledge."<br><b>Implication for reliability:</b><br>Suppose a per-link reliable protocol guarantees each hop delivers frames correctly. The end-to-end TCP connection still needs to check for errors because:<br>• Routers may drop packets under congestion (no per-link reliability at IP layer helps)<br>• Software bugs in router OS could corrupt data<br>• The final application must verify correctness regardless<br>So per-link reliability is wasted effort — end-to-end TCP catches what link-level protocols cannot.<br><b>Why IP is simple/best-effort:</b> Complex functions (reliability, ordering, flow control) are all at endpoints. IP core only does datagram routing — easy to scale, easy to build new hardware for.<br><b>Violations:</b> NAT (routers maintaining flow state), firewalls (deep packet inspection), TCP proxies in CDNs all violate the principle — trading end-to-end transparency for practical deployment benefits.
What is virtual circuit switching vs. datagram switching — mechanically?	<b>Datagram (IP Internet):</b><br>• Each packet carries full destination address<br>• Each router independently looks up dest address in forwarding table (longest prefix match)<br>• Different packets of same flow may take different paths<br>• No setup, no per-flow state in routers<br>• Resilient: if router fails, packets rerouted automatically<br><b>Virtual Circuit (ATM, MPLS):</b><br>• Setup phase: connection established, each switch along path allocated a VCI (Virtual Circuit Identifier)<br>• Each switch has VC table: (input_port, VCI_in) → (output_port, VCI_out)<br>• Data: each packet carries short VCI (not full address) — faster lookup (exact match vs. LPM)<br>• Ordered delivery guaranteed (one path)<br>• QoS possible: resources reserved during setup<br>• Complex failure handling: must re-establish VC on failure<br><b>Switched VC:</b> Set up on-demand (like a phone call)<br><b>Permanent VC:</b> Provisioned by network admin; always up (like a leased line)<br><b>MPLS is a hybrid:</b> VC forwarding semantics over IP/datagram infrastructure.
Compare DiffServ (per-class) vs. IntServ/RSVP (per-flow) QoS.	<b>IntServ (Integrated Services) with RSVP:</b><br>• Per-flow reservations: every router along the path stores state for every individual flow<br>• Hard QoS guarantees: exact bandwidth, delay bound promised to each flow<br>• Scalability: O(flows) state per router. 1M flows in core = 1M state entries. Infeasible for Internet core.<br>• Still used in: MPLS-TE (traffic engineering within ISP), enterprise WAN<br><b>DiffServ (Differentiated Services):</b><br>• Per-class treatment: classify and mark at edge; core routers only look at DSCP field<br>• Scalability: O(classes) state per router (typically &lt;64 DSCP values) — constant regardless of flow count<br>• Relative QoS: "EF traffic gets low delay" — not a hard per-flow guarantee<br>• Simple to implement: each router only needs a priority queue per class<br>• Deployed in: most enterprise networks and ISPs (VoIP marked EF, video marked AF)<br><b>Practical deployment:</b> Edges police/shape/mark traffic; core routers trust DSCP markings and apply PHBs. No per-flow state in core ever.
How does P2P (Gnutella) flooding work and what are its scalability limits?	<b>Gnutella protocol:</b><br><b>JOIN:</b> New node contacts any known Gnutella peer; peer introduces node to the overlay graph.<br><b>QUERY flood (searching):</b><br>1. Node A generates QUERY message with QID (unique Query ID), search string, TTL<br>2. A sends QUERY to all its neighbors<br>3. Each neighbor: if QID seen before → discard (loop prevention). If not seen: record QID, forward to all its neighbors (except origin), decrement TTL.<br>4. If TTL reaches 0 → stop forwarding<br>5. Any node with matching content sends QUERYHIT back along the reverse path (each node remembered which neighbor forwarded the QUERY to it)<br><b>DATA TRANSFER:</b> Direct TCP/HTTP connection between querier and match — not through the overlay path.<br><b>Scalability issues:</b><br>• Each QUERY generates O(N) messages (flood to all reachable nodes)<br>• High TTL (default 7) → massive traffic<br>• Rare content may never be found (TTL too low, content owner unreachable)<br>• DHT-based systems (Chord, Pastry, Kademlia) solve scalability: O(log N) hops per lookup, guaranteed reachability.
How does BitTorrent's tit-for-tat mechanism incentivize sharing?	<b>Core problem:</b> P2P only works if peers upload. A rational peer would rather just download ("leech").<br><b>Choking algorithm (every 10 seconds):</b><br>1. Rank all neighbors by their upload rate to you (bytes/sec received from them recently)<br>2. "Unchoke" the top N (typically 4) uploaders → you will upload to them<br>3. Choke everyone else → refuse to upload<br><b>Optimistic unchoking (every 30 seconds):</b><br>4. Randomly unchoke one additional peer (the "optimistic unchoke")<br>5. This gives new peers (nothing to trade yet) a chance to receive data<br>6. If they upload well → they'll rank into the regular top-N; otherwise get choked again<br><b>Rarest-first piece selection:</b><br>• Track which pieces are least replicated across the swarm<br>• Download rarest pieces first → improves piece diversity across swarm<br>• As swarm diversity improves, all peers can become seeds faster<br><b>End-game mode:</b> When a peer has all pieces except the last few → request remaining pieces from ALL peers simultaneously → ensures fast completion even if one peer is slow.
What is the difference between routing and forwarding?	<b>Routing (control plane):</b> The background process of computing paths and building forwarding tables. Runs routing protocols (OSPF, BGP, RIP). Happens on timescales of seconds to minutes. Output: a forwarding table mapping destinations to output interfaces/next-hops.<br><b>Forwarding (data plane):</b> The per-packet action of receiving a packet on one interface and sending it out another based on the forwarding table. Happens at line rate (nanoseconds to microseconds). Output: packet delivered to correct next-hop.<br><b>SDN separates them explicitly:</b> Control plane runs as software on controller (commodity server). Data plane runs on bare-metal switch hardware. Southbound API (OpenFlow) connects them.<br><b>Traditional vs. SDN:</b><br>Traditional: Tightly coupled — each router runs both routing protocols and forwarding in the same box. Changes require per-device updates.<br>SDN: Controller has global network view; can compute globally optimal forwarding rules and push them to all switches simultaneously.
What is RED (Random Early Detection) and how does it prevent global synchronization?	<b>Problem with Tail Drop:</b> Under high load, router buffer fills → ALL arriving packets dropped simultaneously → ALL TCP senders simultaneously detect loss → ALL reduce cwnd at same time → Link goes from 100% utilization to near-0% (underutilized) → Everyone slow-starts again → Oscillation. This is "global synchronization."<br><b>RED algorithm:</b><br>Router maintains EWMA of queue length: avg_q = (1-w)×avg_q + w×q (w small, e.g., 0.002)<br>1. avg_q < min_th: accept all packets<br>2. avg_q > max_th: drop all packets (tail drop)<br>3. min_th ≤ avg_q ≤ max_th: drop with probability P = max_p × (avg_q − min_th)/(max_th − min_th)<br><b>Why randomness prevents synchronization:</b><br>• With high load, many flows contribute packets. RED statistically drops more from heavy flows (they send more packets → more chances to be dropped).<br>• Different flows' packets are dropped at different random times → they enter Multiplicative Decrease at different moments → no synchronized sawtooth → link stays better utilized.<br><b>ECN + RED:</b> Replace drops with CE marking → same staggered signal, zero retransmission overhead.
What are the DNS record types and when is each used?	DNS records are 5-tuples: (Name, TTL, Class, Type, Value). Class is nearly always IN (Internet).<br><b>A:</b> Maps hostname → IPv4 address. "www.example.com → 93.184.216.34"<br><b>AAAA:</b> Maps hostname → IPv6 address. "www.example.com → 2606:2800:..."<br><b>NS:</b> Maps zone → authoritative name server hostname. "example.com NS ns1.example.com". Used during iterative resolution.<br><b>CNAME:</b> Alias → canonical name. "www → example.com". Resolver must follow and resolve the canonical name. Cannot exist at zone apex.<br><b>MX:</b> Domain → mail exchanger hostname + priority. "example.com MX 10 mail.example.com". Lower priority number = preferred.<br><b>PTR:</b> Reverse lookup: IP → hostname (used by spam filters, syslog). "34.216.184.93.in-addr.arpa → www.example.com"<br><b>SOA:</b> Start of Authority — zone serial, refresh, retry, expire, minimum TTL, primary NS, admin email. One per zone.<br><b>TXT:</b> Free text — used for SPF (email sender policy), DKIM public keys, domain ownership verification, DMARC.<br><b>TTL:</b> How long (seconds) resolvers may cache this record before re-querying.
How does DNSSEC protect DNS and what is the chain of trust?	<b>Problem:</b> DNS responses are unauthenticated — a Kaminsky-style attack can poison a recursive resolver's cache, redirecting all users querying that resolver to a malicious IP.<br><b>DNSSEC solution — cryptographic signing:</b><br>1. Zone owner generates public/private keypair (ZSK = Zone Signing Key)<br>2. Signs each RRset (group of records of same type/name) → produces RRSIG record<br>3. Publishes DNSKEY record (public key) in the zone<br><b>Chain of trust (from root to leaf):</b><br>1. Root zone signs its DNSKEY with its own private key<br>2. Root signs the .com TLD's DNSKEY hash → DS (Delegation Signer) record<br>3. .com zone signs its DNSKEY; signs example.com's DNSKEY hash → DS record<br>4. example.com zone signs its own records with its ZSK<br><b>Validation:</b> Resolver fetches RRSIG, verifies signature using DNSKEY; verifies DNSKEY against parent's DS record; chain up to root (root's DNSKEY is pre-configured as trust anchor).<br><b>Limitation:</b> DNSSEC authenticates but does NOT encrypt DNS. Anyone can read DNS queries/responses (use DNS-over-HTTPS/TLS for privacy).
How does HTTP caching with ETags and Cache-Control work mechanically?	<b>First request:</b><br>→ GET /style.css HTTP/1.1<br>← HTTP/1.1 200 OK<br>← Cache-Control: max-age=3600<br>← ETag: "abc123"<br>← [full CSS body]<br>Browser caches CSS with ETag="abc123" and a freshness deadline of now+3600s.<br><b>Subsequent request BEFORE expiry (0 RTTs):</b><br>Browser serves cached CSS directly. No network request needed.<br><b>Request AFTER expiry (conditional GET — 1 RTT):</b><br>→ GET /style.css HTTP/1.1<br>→ If-None-Match: "abc123"<br>Server computes ETag of current file:<br>If ETag unchanged: ← 304 Not Modified (no body) — browser uses cached copy.<br>If ETag changed: ← 200 OK + new ETag + new CSS body — browser updates cache.<br><b>Cache-Control directives:</b><br>• max-age=N: fresh for N seconds<br>• no-store: never cache<br>• no-cache: must revalidate before every use<br>• private: browser-only (not CDN/proxy)<br>• public: any intermediate can cache<br>• must-revalidate: if stale, don't use without server confirmation
How does an SDN OpenFlow pipeline process a packet?	<b>Packet arrives at OpenFlow switch:</b><br>1. Parser extracts header fields: src/dst MAC, src/dst IP, VLAN ID, IP protocol, TCP/UDP src/dst ports, MPLS label, etc.<br>2. Match against flow table entries in priority order (highest priority number first)<br>3. First matching entry found → execute its Action Set:<br>   • Output(port): forward out specific port; Output(FLOOD): all ports; Output(CONTROLLER): send to controller<br>   • Drop: discard<br>   • Set-Field: modify any header field (rewrite dst IP, push/pop VLAN, change DSCP, decrement TTL)<br>   • Group: multicast, fast-failover, or ECMP<br>4. No match → table-miss flow: typically send to controller (Packet-In message) or drop<br><b>Reactive flow installation:</b><br>• First packet of a new flow → Packet-In sent to controller<br>• Controller decides: block, forward, redirect, monitor<br>• Controller sends FlowMod → installs rule in switch<br>• Subsequent packets of same flow hit the installed rule (no controller involvement)<br><b>Proactive installation:</b> Controller pre-installs rules before traffic arrives (lower latency, predictable behavior).
How does PON handle the upstream collision problem (multiple ONUs sharing fiber)?	<b>Problem:</b> Multiple ONUs (homes) share the same upstream fiber wavelength. If two ONUs transmit simultaneously their laser pulses combine at the splitter and become unreadable at the OLT.<br><b>Solution: OLT-controlled TDMA</b><br>1. OLT is the only entity that schedules upstream transmission<br>2. OLT sends GATE messages granting each ONU a specific transmission window (start time + duration)<br>3. ONUs transmit ONLY during their allocated window — never spontaneously<br><b>Ranging (initialization of new ONU):</b><br>1. OLT opens a quiet window (all existing ONUs silent)<br>2. OLT sends Ranging Request to new ONU<br>3. ONU transmits ranging response<br>4. OLT measures round-trip time (propagation delay varies by ONU distance from splitter)<br>5. OLT assigns an equalization delay to the ONU: ONUs further away transmit earlier to compensate → all ONUs appear equidistant to OLT<br><b>DBA (Dynamic Bandwidth Allocation):</b><br>ONUs report queue depth → OLT dynamically adjusts grants to match load. Idle ONUs get small grant; busy ONUs get more bandwidth. Avoids fixed TDM waste.
How does anycast routing work for DNS root servers?	<b>Anycast:</b> Multiple physical servers share the same IP prefix/address. BGP announces the prefix from each server location. Packets are routed to the "nearest" (shortest BGP path) server.<br><b>DNS root server deployment:</b><br>• 13 logical root names (a–m.root-servers.net), each with one IPv4 address<br>• In reality: 600+ physical server instances worldwide across many data centers<br>• Each instance announces its /32 (or small prefix) to local upstream ISPs via BGP<br>• BGP naturally routes queries to the geographically/topologically closest instance<br><b>Benefits:</b><br>• Geographic distribution → low latency for resolvers worldwide<br>• DDoS resilience → attack traffic is absorbed by nearest instance; global system unaffected<br>• High availability → if one instance fails, BGP reconverges, queries route to next-nearest<br><b>Anycast + TCP limitation:</b><br>Anycast is safe for DNS (UDP, stateless per-query) but problematic for long-lived TCP connections (mid-connection routing change could send packets to a different server instance that has no state for the connection).
What is VPN tunneling and how does IPsec encapsulate packets?	<b>Scenario:</b> Two branch offices connected over the public Internet with an encrypted tunnel.<br><b>IPsec Tunnel Mode (the standard for site-to-site VPN):</b><br>Outbound packet flow (Host A: 10.1.0.5 → Host B: 10.2.0.5):<br>1. Host A sends normal IP packet: src=10.1.0.5, dst=10.2.0.5<br>2. Branch A's VPN gateway intercepts (10.2.0.0/24 in VPN policy)<br>3. Encrypts entire original IP packet (header + payload) using ESP (Encapsulating Security Payload): AES-256-GCM<br>4. Adds new IP header: src=203.0.113.1 (Branch A public IP), dst=198.51.100.1 (Branch B public IP)<br>5. Adds ESP trailer with authentication tag (HMAC)<br>6. Packet traverses public Internet — looks like encrypted UDP/50 traffic between two public IPs<br><b>Inbound at Branch B's VPN gateway:</b><br>7. Receive outer IP packet; identify as IPsec (protocol 50 = ESP)<br>8. Verify authentication tag; decrypt using SA (Security Association) shared keys<br>9. Restore original inner packet: src=10.1.0.5, dst=10.2.0.5<br>10. Route to Host B on the local LAN<br><b>IKEv2:</b> Key exchange protocol that establishes the SA (encrypts SA negotiation itself).
What is the difference between flow control and congestion control?	<b>Flow Control (point-to-point, TCP rwnd):</b><br>• Prevents a fast sender from overwhelming a <i>slow receiver's buffer</i><br>• Signal: receiver's buffer occupancy → advertised in TCP Window field of every ACK<br>• Scope: one TCP connection, one receiver<br>• Mechanism: sender limits in-flight data to receiver's announced rwnd<br>• No network visibility: only receiver's state matters<br><b>Congestion Control (network-wide, TCP cwnd):</b><br>• Prevents senders from collectively overwhelming <i>shared network links and router buffers</i><br>• Signal: packet loss (implicit) or ECN marks (explicit) or increasing RTT (delay-based)<br>• Scope: all senders competing for the same bottleneck link<br>• Mechanism: sender limits in-flight data to cwnd (maintained by sender, adjusted based on network feedback)<br>• No direct visibility into router queues: inferred from observable signals<br><b>Interaction:</b> TCP's effective sending rate = min(cwnd, rwnd) / RTT<br>Both are necessary: flow control protects receivers; congestion control protects the network.
What is Maximum Segment Size (MSS) and how is it determined?	**Maximum Segment Size (MSS)** is the largest amount of data, specified in bytes, that a **[[Transmission Control Protocol (TCP)|Transmission Control Protocol (TCP)]]** endpoint is willing to receive in a single segment.<br><b>Determination:</b><br>1. It is negotiated during the **[[CLAUDE#Back|Three-Way Handshake]]** using a header option.<br>2. It is typically set to the **[[Maximum Transmission Unit (MTU)|Maximum Transmission Unit (MTU)]]** of the local outgoing link minus the sizes of the **[[Internet Protocol (IP)|Internet Protocol (IP)]]** and **[[Transmission Control Protocol (TCP)|TCP]]** headers (usually 1500 - 20 - 20 = 1460 bytes for Ethernet).<br><b>Purpose:</b> Prevents **[[IP Fragmentation and Reassembly|IP fragmentation]]** at the network layer by ensuring segments fit within the link-layer frames.
What is Selective Acknowledgement (SACK) and why is it superior to cumulative ACKs?	**Selective Acknowledgement (SACK)** is a **[[Transmission Control Protocol (TCP)|Transmission Control Protocol (TCP)]]** option that allows a receiver to inform the sender about all segments that have been received successfully, including non-contiguous blocks.<br><b>Superiority:</b><br>1. **Cumulative Acknowledgements (ACKs)** only inform the sender of the last in-order byte received; if multiple segments are lost, the sender may have to wait for a **[[How does the Jacobson/Karels adaptive RTO work step by step?|Retransmission Timeout (RTO)]]** or retransmit everything from the point of loss.<br>2. **SACK** allows the sender to retransmit <i>only</i> the specific missing segments in a single **[[Round-Trip Time (RTT)|Round-Trip Time (RTT)]]**, significantly improving throughput in high-loss environments.
What is the Window Scale option and why is it necessary for high-speed networks?	The **Window Scale** option is a **[[Transmission Control Protocol (TCP)|Transmission Control Protocol (TCP)]]** extension that increases the maximum allowed size of the receive window from 64 KB to approximately 1 GB.<br><b>Necessity:</b><br>1. The standard **[[Advertised Window (rwnd)|Advertised Window]]** field in the header is only 16 bits, limiting the window to $2^{16} - 1 = 65,535$ bytes.<br>2. In high-speed, high-latency networks (high **[[Bandwidth-Delay Product (BDP)|Bandwidth-Delay Product (BDP)]]**), a 64 KB window is too small to keep the pipe full, as the sender must stop and wait for **[[Acknowledgment (ACK)|Acknowledgements (ACKs)]]** before sending more data.<br>3. **Window Scale** provides a scale factor (shift count) negotiated during the handshake to multiply the header's window value.
What do Transmission Control Protocol (TCP) sequence numbers count — bytes or packets?	They count **bytes**.<br>TCP treats data as a continuous **[[Transport Layer - Transmission Control Protocol (TCP)#Low-Level Primer: The Reliable Byte-Stream|Reliable Byte-Stream]]**, not a sequence of individual packets. The sequence number in a segment header represents the byte-stream index of the <i>first</i> data byte carried in that segment.
What does an Acknowledgment (ACK) number mean exactly?	The **Acknowledgment (ACK)** number is the **Next Expected Byte** index from the peer.<br>It is a **cumulative ACK**, meaning that all bytes up to (but not including) that number have been successfully received and reassembled. For example, if a receiver sends an ACK with number 1001, it is telling the sender: "I have received everything up to byte 1000; please send byte 1001 next."
How does Transmission Control Protocol (TCP) detect duplicate packets?	It uses **sequence numbers**.<br>If a segment arrives with a sequence number that has already been received and acknowledged (or falls below the current expected sequence number), the receiver identifies it as a duplicate and discards the payload while typically sending a new **[[Acknowledgment (ACK)|Acknowledgment (ACK)]]** to the sender to synchronize state.
How does Transmission Control Protocol (TCP) detect reordered packets?	It uses **sequence numbers** and the **[[Advertised Window (rwnd)|Receive Window]]** buffer.<br>1. If a segment arrives with a sequence number higher than the next expected byte, the receiver knows a gap exists (reordering or loss).<br>2. The receiver places the out-of-order segment into its buffer (within the window) and sends a **[[Acknowledgment (ACK)|Acknowledgment (ACK)]]** for the <i>last in-order</i> byte received.<br>3. Once the missing bytes arrive, the receiver "fills the gap" and can deliver the contiguous data to the application.
What is a cumulative Acknowledgment (ACK)?	A **cumulative Acknowledgment (ACK)** is a single acknowledgement that covers all data received up to a certain point in the stream.<br>Instead of acknowledging each individual segment, the **[[Transmission Control Protocol (TCP)|Transmission Control Protocol (TCP)]]** **ACK Number** field always specifies the next byte the receiver is waiting for. This provides robustness: if an ACK is lost, a subsequent ACK for a later byte implicitly acknowledges all the data that the lost ACK would have covered.
How is the Selective Acknowledgement (SACK) option negotiated and what does it look like in the header?	**Negotiation:**<br>It is enabled during the **[[CLAUDE#Back|Three-Way Handshake]]**; both sides must include a `SACK-Permitted` option in their `SYN` and `SYN-ACK` segments.<br><b>Header Appearance:</b><br>When used in an **[[Acknowledgment (ACK)|Acknowledgment (ACK)]]**, the option contains a list of **SACK Blocks**. Each block consists of a **Left Edge** and a **Right Edge** (32-bit sequence numbers) defining a contiguous range of bytes that have been received out-of-order.
What is a duplicate Acknowledgment (ACK)?	A **duplicate Acknowledgment (ACK)** is an **[[Acknowledgment (ACK)|Acknowledgment (ACK)]]** that carries the exact same **ACK Number** as a previously sent/received ACK, usually sent in response to an out-of-order segment.<br>While the ACK number doesn't advance, the segment may still carry a **[[Selective Acknowledgement (SACK)|Selective Acknowledgement (SACK)]]** block or simply serve as a signal to the sender that "I am still waiting for byte X, but I just received something else."
What does it mean when the sender receives three duplicate Acknowledgements (ACKs)?	It is treated as a strong signal of **segment loss** (not just simple reordering).<br><b>Action:</b> The sender performs **[[Transport Layer - TCP Congestion Control#3. Loss Detection and Recovery|Fast Retransmit]]**, immediately resending the missing segment indicated by the ACK number without waiting for the **[[How does the Jacobson/Karels adaptive RTO work step by step?|Retransmission Timeout (RTO)]]** to expire. It then enters **Fast Recovery** to adjust its **[[Congestion Window (cwnd)|Congestion Window (cwnd)]]**.
What is a zero window and what happens when the sender receives one?	A **zero window** occurs when a receiver advertises an **[[Advertised Window (rwnd)|Advertised Window]]** of 0 bytes, signaling that its internal buffer is completely full.<br><b>Sender Action:</b><br>1. The sender must **stop transmitting** new data immediately.<br>2. It starts a **persist timer** to periodically "probe" the receiver.<br>3. This prevents a deadlock if the "window open" update from the receiver is lost.
What is a window probe?	A **window probe** is a 1-byte segment sent by a **[[Transmission Control Protocol (TCP)|Transmission Control Protocol (TCP)]]** sender when the receiver has advertised a **zero window**.<br><b>Purpose:</b> It forces the receiver to respond with an **[[Acknowledgment (ACK)|Acknowledgment (ACK)]]** that includes the current **[[Advertised Window (rwnd)|Advertised Window]]** size. This allows the sender to discover when the receiver's buffer has cleared and it can resume transmission, protecting against the loss of the receiver's unsolicited window update.
What is a persist timer?	The **persist timer** is a **[[Transmission Control Protocol (TCP)|Transmission Control Protocol (TCP)]]** timer used by the sender to manage the **zero window** state.<br><b>Mechanism:</b><br>1. When the sender receives an **[[Acknowledgment (ACK)|Acknowledgment (ACK)]]** with a window size of 0, it starts the persist timer.<br>2. When the timer expires, the sender sends a **window probe**.<br>3. If the response still shows a zero window, the timer is reset (usually with **[[Binary Exponential Backoff (BEB)|Binary Exponential Backoff (BEB)]]**) and the process repeats until the window opens.
